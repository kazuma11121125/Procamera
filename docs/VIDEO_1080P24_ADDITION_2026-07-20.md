# 1080p24 動画設定候補の追加 — 実機検証記録

最終更新: 2026-07-20
コード基準: `main`(`c1613f0`) + branch `feat/add-1080p24`
対象実機: Sony SO-51C

## 目的

`docs/VIDEO_FPS_STUTTER_INVESTIGATION_2026-07-20.md`の調査で、
`CONTROL_AE_AVAILABLE_TARGET_FPS_RANGES`に`[24, 24]`が実在することが分かった。
30fpsよりさらに余裕を持たせたfpsを選べるよう、`1080p24`(AVC, 8Mbps)を
`CameraCapabilityInspector.supportedVideoConfigs()`の候補に追加した。

**この変更は原因②(高温時のHAL内部フレーム損失)の根本修正ではない。** 単に
選択肢を1つ増やすだけで、原因②自体は未解決のまま。

## 変更

`CameraCapabilityInspector.kt`の`allCandidates`に1行追加のみ:

```kotlin
VideoConfigCandidate(MediaFormat.MIMETYPE_VIDEO_AVC, 1920, 1080, 24, 8_000_000), // 16:9
```

ビットレートは1080p30(10Mbps)からfps比(24/30)で単純に落とした。

## 実機検証(すべて確認済み)

| 項目 | 結果 |
|---|---|
| 1080p24 / AUTO | 設定画面に表示・選択可能。`ManualCaptureRequestFactory`のログで`target=24, selected=(24,24)`を確認(=`CONTROL_AE_AVAILABLE_TARGET_FPS_RANGES`の`[24,24]`固定レンジが正しく選ばれている) |
| 1080p24 / MANUAL | 選択・録画可能 |
| MANUAL露光時間のクランプ | `CaptureResult.SENSOR_EXPOSURE_TIME`を実機で直接確認したところ`41,641,000ns`(≤ 1/24s = 41,666,667ns)——HALへ渡る値は正しくクランプされている。**UI表示についても再検証済み**: 1080p60でSHUTTERを1/1秒まで遅くしてから1080p24へ切り替えると、表示が即座に「1/24」へ更新されることを実機で確認(初回レビュー時に「UIが古い値を表示したままだった」と報告したのは筆者の見間違いで、再現しなかった) |
| AUTO時のfpsレンジ選択 | `[24,24]`固定レンジが選ばれることをログで確認 |
| MP4のfps指標(下記「fps指標の内訳」参照) | コンテナ宣言(`r_frame_rate`)23.833fps、PTS間隔の中央値23.84fps——両者は一致。単純平均(`nb_frames/duration`)21.21fpsは低いが、これは後述のスタッター(主に録画開始直後)による既知の影響で説明でき、未説明のズレはない |
| 録画停止・再開 | 正常 |
| 再起動後の設定復元 | 1080p24 + ExposureModeともに正しく復元(ログで`target=24, selected=(24,24)`を再確認) |
| レンズ切替 | **1080p24とは無関係の既存動作** として確認: 手動でのレンズタップは`switchLens(lens)`(`initialVideoConfig`なし)を呼び、常にデフォルト(最大解像度)へリセットする。`git diff main`でこのパスに差分がないことを確認済み——今回のPRの範囲外(別Issueとしてユーザーへ報告済み、次のタスクで対応予定) |

### fps指標の内訳(常温録画、`AuCamPRO_20260720_231330_864.mp4`、約10.65秒)

`rec_diagnose.py`の「宣言fps」は`ffprobe`の`r_frame_rate`(コンテナ/ストリームの基準
レート)であり、`avg_frame_rate`でも単純平均でもない——後者は原因①の調査で「スタッター
した分だけ基準が緩んで検出しづらくなる」問題があったため、あえて避けている
(docs/VIDEO_FPS_STUTTER_INVESTIGATION_2026-07-20.md §2参照)。この録画について
3つの指標を分けて実測した:

| 指標 | 意味 | 値 |
|---|---|---:|
| `r_frame_rate` | コンテナ/ストリーム上の基準レート(`143/6`) | 23.8333fps |
| 単純平均(`nb_frames/duration`) | 総フレーム数226÷実測時間——スタッターの影響を直接含む | 21.21fps |
| PTS間隔の中央値 | 「普段」の1フレームあたり間隔——外れ値(スタッター)に強い | 23.84fps |

`r_frame_rate`(23.833)と中央値(23.84)はほぼ一致——24fps自体は正しく機能している。
単純平均(21.21)だけが低いのは、下記のスタッターで説明できる範囲内で、24fps特有の
未説明のズレではない:

- 最大の遅延は録画開始直後(0.29〜0.92秒、2件で計約920ms)——既存の録画開始時セッション
  再構成コスト(実機計測で~650-1000ms、既知)と一致。前半(7.14%)>後半(1.77%)で
  熱による悪化パターンではない
- 残りは散発的な単発フレーム遅延(83.9ms=期待値の約2倍が6件)。24fps特有の新しい問題では
  なさそうだが、完全にゼロではない
- スタッター率4.4%(10件)、失われた時間合計1216.4ms(全体の約11.4%)——この分だけ単純
  平均が中央値/宣言値より低く出ることと整合する

### やや高温状態での録画(1080p24 / AUTO、約25秒)

`CAM_CRITICAL`には到達していない中程度の熱量。

| 指標 | 値 |
|---|---:|
| `StreamingA`(センサー) | 23.95〜24.16fps(健全) |
| `video stream`/`preview stream`(HAL配信) | 18〜20fps |
| `CAM_CRITICAL`発火 | なし(今回のテストではここまで到達しなかった) |
| スタッター率(ファイル解析) | 12.5%、前半16.1%>後半9.0%(熱による悪化パターンではない) |

センサーと配信の比は約75-83%——`docs/VIDEO_FPS_STUTTER_INVESTIGATION_2026-07-20.md`
§3.2で見たAE_ON/30fpsの比(21-22/29.97 ≈ 70-73%)と同程度かやや良好。ただし**絶対値としては
30fps/60fpsで見た深刻な崩壊(3-5fps)よりずっと軽い**——単純にセンサーへの要求(24fps)自体が
低いため、同じ相対的な損失率でも実効fpsが高く出ている、という程度の効果と考えるのが妥当。

**真の`CAM_CRITICAL`状態での1080p24の挙動は今回未確認。** 原因②の実用的な回避策になるか
どうかは、`CAM_CRITICAL`発火を確認できる熱量での追加検証が必要。

## まとめ

1080p24候補の追加自体は実機で問題なく動作する(設定・録画・復元すべて正常)。原因②への
効果は限定的・未確定(センサー要求が低い分、同じ相対損失率でも絶対fpsが高く出るだけ)——
根本修正としては扱わない。次の作業は別スコープ:

1. レンズ切替時に解像度がリセットされる既存動作の改善(別PR)
2. 原因②本体: AE_ON / AE_ON+AE_LOCK / AE_OFFの3条件比較(別ブランチ、`CAM_CRITICAL`確定下)
